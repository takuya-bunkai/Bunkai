🔍 Workflow Data Fabric (WDF) とは？

ServiceNowが真のAI Platformとなるために、従来Now Platformに保存されたデータだけでなく、ServiceNow以外にあるデータにもシームレスにアクセスし、AIエージェントが活用できる構想がWDFです。

古くは Import Set、スクリプティングから始まった外部連携は、Orchestration や Integration Hub を経て Automation Hub へと進化してきましたが、WDF ではさらに以下の強化が期待されます：

  • 接続先の拡大
  • リアルタイム性の向上
  • データ連携の柔軟性
  • 開発者にとっての扱いやすさ
  • セキュリティ／ガバナンスの担保

🔄 これまでのデータ連携における代表的な課題

現場でよく聞く「これ、またか…」という悩み、実は多くの組織が共通して抱えてきたものでした。セッションでも強調されていた、代表的な 4 つの課題をご紹介します：

  1. 高いメンテナンス負荷 (High Maintenance)
    データが複製されることで、「信頼できる唯一の情報源（single source of truth）」が失われ、整合性チェックや同期の手間が増加します。
  2. データ遅延 (Data Latency)
    大容量データを別システムに移す際の転送遅延やバッチ処理の待ち時間により、後続タスクの自動実行や処理着手も滞るため、意思決定だけでなくビジネス全体の流れが鈍化します。
  3. 固定的なマッピング構造 (Rigid Data Mapping)
    実行時の情報に対する柔軟な対応が求められる中、事前定義されたマッピングでは対応が難しく、設計変更のたびに工数が発生します。
  4. 自動化の限界 (Limited Automation)
    多くの従来の連携は一方向またはバッチ実行に依存しており、リアルタイム双方向の自動化には向いていません。

その中でも特に、Zero Copy Connectors は「データを動かさずに使う」ための鍵となる機能です。

🧩 構成図内の主な要素とその意味

Off-Instance Processing： ServiceNowインスタンスの負荷を軽減するため、データ処理をServiceNow外部で行う構造です。これにより、大量データ処理や分析を高速かつ効率的に行えるようになります。AI推論や大規模なクエリ処理に有効です。

Regional Cluster： データ処理は、ServiceNowのインスタンスと地理的に近いクラスタで実行されます。これにより、ネットワーク遅延を抑えつつ、セキュアな環境でのフェデレーテッドクエリ実行が実現されます。

Translated to SQL Query： アプリケーション開発者が記述する GlideRecord API でのデータアクセスは、内部的にSQLクエリへと変換され、外部データソースに対して実行されます。開発者は「従来通り」の書き方で、実は外部データにアクセスできているというわけです。

Federated Query Engine (Trino)： SQLに変換されたクエリは、Trinoをベースとした分散型のフェデレーテッドクエリエンジンを通じて実行されます。これにより、複数のデータソース（Snowflake, BigQueryなど）に対して統一的なインターフェースでアクセスが可能になります。

Extensible Connector Framework： 各種外部データソースとの接続は、拡張可能なコネクターフレームワークを通じて行われます。これにより、Snowflake や Databricks 以外にも、今後の接続対象が増えることが期待されています。

Zero Copy Connectors： データをServiceNowにコピーすることなく、参照だけで活用できる仮想テーブルの仕組みです。これにより、ストレージ節約・リアルタイム性・セキュリティポリシーの一貫性が保たれます。構成図上では、上記の要素すべてをつなぐ中心的な役割を果たしています。
